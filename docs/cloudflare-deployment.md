[{"type":"text","text":"Think of production stateful agents as needing four things:\n\n1. **A request edge** — fast auth/routing/streaming.\n2. **A state kernel** — durable per-user/per-conversation memory and coordination.\n3. **A durable executor** — something that survives long runs, retries, sleeps, human approval.\n4. **External stores** — vector memory, blobs, relational metadata, traces, logs.\n\nCloudflare gives you most of this natively, but you need to map LangChain/LangGraph concepts onto Cloudflare primitives carefully.\n\n---\n\n# 1. The core production architecture\n\nA strong Cloudflare-native architecture looks like this:\n\n```text\nClient / API / Slack / Webhook\n        |\n        v\nCloudflare Worker\n- auth\n- rate limiting\n- request validation\n- streaming response\n- routes requests\n        |\n        v\nDurable Object per agent/conversation/user\n- stateful agent session\n- message ordering\n- short-term memory\n- locks/idempotency\n- WebSocket/SSE coordination\n- local SQLite state\n        |\n        +--> Cloudflare Workflow\n        |    - long-running agent tasks\n        |    - retries\n        |    - sleeps\n        |    - human-in-the-loop waits\n        |\n        +--> Queue\n        |    - async tools\n        |    - embedding jobs\n        |    - summarization\n        |    - ingestion\n        |\n        +--> Vectorize\n        |    - semantic memory\n        |    - RAG search\n        |\n        +--> R2\n        |    - files\n        |    - transcripts\n        |    - artifacts\n        |    - large tool outputs\n        |\n        +--> D1 / external Postgres\n        |    - users\n        |    - runs\n        |    - audit records\n        |    - billing\n        |    - agent configs\n        |\n        +--> AI Gateway\n             - OpenAI/Anthropic/etc.\n             - Workers AI\n             - logs\n             - retries\n             - rate limits\n             - caching\n             - fallback\n```\n\nThe short version:\n\n> **Workers handle requests. Durable Objects hold live state. Workflows run long tasks. Queues decouple expensive work. R2 stores large artifacts. D1/Postgres stores metadata. Vectorize stores semantic memory. AI Gateway wraps model calls.**\n\n---\n\n# 2. Where LangChain fits\n\nThere are two different paths.\n\n## Path A: Cloudflare-native TypeScript agents\n\nUse:\n\n- Cloudflare Workers\n- Durable Objects\n- Workflows\n- Queues\n- LangChain.js\n- AI Gateway\n- Workers AI or external LLMs\n\nThis is good if you want a mostly edge-native TypeScript stack.\n\nExample use case:\n\n```text\nUser sends chat message\n   |\nWorker receives request\n   |\nRoutes to Durable Object for conversation_id\n   |\nDurable Object loads state\n   |\nLangChain.js agent runs one turn\n   |\nModel call goes through AI Gateway\n   |\nState is saved\n   |\nResponse streams back\n```\n\nThis works well for bounded chat-style agents.\n\nBut there are caveats:\n\n- Not all LangChain packages are edge-compatible.\n- Workers do not support a normal long-lived Node.js process.\n- Native Node modules, filesystem access, child processes, raw TCP assumptions, or heavy browser tooling may not work.\n- Complex Python LangGraph/deepagents do not naturally run inside Workers.\n\nSo this is best for **JS/TS agents with relatively controlled execution**.\n\n---\n\n## Path B: Cloudflare as edge gateway, LangGraph elsewhere\n\nFor serious LangChain/LangGraph/deepagents production systems, this is often better:\n\n```text\nClient\n   |\nCloudflare Worker\n- auth\n- rate limit\n- abuse protection\n- streaming proxy\n   |\nRegional LangGraph/LangSmith Agent Server\n- Python or Node agent runtime\n- LangGraph checkpointing\n- deepagents\n- tools\n- human-in-the-loop\n   |\nPostgres / Redis / object storage / vector DB\n```\n\nCloudflare still does what it is excellent at:\n\n- global routing\n- security\n- rate limiting\n- low-latency ingress\n- streaming proxy\n- AI Gateway\n- R2\n- Vectorize\n- Workers AI\n- Queues\n\nBut the full agent runtime runs in something like:\n\n- LangGraph Platform\n- LangSmith Agent Server\n- Fly.io\n- Cloud Run\n- ECS/Fargate\n- Kubernetes\n- Railway/Render\n- your own container platform\n\nThis is better if you need:\n\n- Python LangGraph\n- deepagents\n- browser automation\n- code execution\n- long-running research agents\n- complex tools\n- filesystem-like workspaces\n- heavy dependencies\n- reliable checkpointing with Postgres\n\n---\n\n# 3. Cloudflare primitives mapped to agent concepts\n\n## Worker = stateless API edge\n\nUse Workers for:\n\n- auth\n- request validation\n- routing\n- rate limiting\n- tenant lookup\n- starting or resuming agent runs\n- streaming responses\n- calling Durable Objects, Queues, Workflows, R2, D1, Vectorize\n\nDo **not** treat Workers as your durable agent memory.\n\nBad:\n\n```ts\nlet conversationMemory = {};\n```\n\nThat may disappear, duplicate, or become inconsistent.\n\nGood:\n\n```text\nWorker receives request\nWorker forwards request to Durable Object\nDurable Object owns state\n```\n\n---\n\n## Durable Object = stateful agent/session kernel\n\nDurable Objects are the most important Cloudflare primitive for stateful agents.\n\nUse one Durable Object per:\n\n- conversation\n- agent instance\n- user session\n- workspace\n- tenant shard\n\nA Durable Object gives you:\n\n- single-threaded coordination\n- durable storage\n- local SQLite-backed state\n- ordered handling for one session\n- WebSocket coordination\n- per-agent locks\n- idempotency tracking\n\nExample conceptual model:\n\n```text\nConversationDurableObject: conversation_abc123\n\nState:\n- messages\n- current_summary\n- active_run_id\n- tool_call_ledger\n- pending_human_approval\n- last_event_index\n```\n\nThis solves one of the hardest production agent problems: **two requests for the same conversation arriving at the same time**.\n\nWithout a Durable Object, you can easily get race conditions:\n\n```text\nUser sends message A\nUser quickly sends message B\n\nBoth read old memory\nBoth call model\nBoth write state\nState becomes inconsistent\n```\n\nWith a Durable Object:\n\n```text\nMessage A handled first\nState updated\nMessage B handled second\nState updated from correct latest state\n```\n\n---\n\n## Workflow = durable long-running agent run\n\nUse Cloudflare Workflows for tasks that may:\n\n- take minutes\n- require retries\n- wait for a human\n- sleep until later\n- perform multiple steps\n- survive failures\n- call external tools\n- produce artifacts\n\nExamples:\n\n```text\n\"Research this company and write a report\"\n\"Monitor this website every hour\"\n\"Analyze these uploaded documents\"\n\"Generate a proposal, then wait for approval\"\n\"Run this multi-step sales outreach workflow\"\n```\n\nDo not keep an HTTP request open for these.\n\nBetter pattern:\n\n```text\nPOST /agents/run\n   |\nCreate workflow\n   |\nReturn run_id immediately\n   |\nClient polls or subscribes to stream\n```\n\n---\n\n## Queue = async buffer\n\nUse Queues for:\n\n- embeddings\n- document ingestion\n- background summarization\n- email sending\n- webhook retries\n- slow tools\n- fanout jobs\n- evaluation jobs\n- model call backpressure\n\nQueues should be idempotent.\n\nEvery message should include something like:\n\n```json\n{\n  \"jobId\": \"job_123\",\n  \"tenantId\": \"tenant_456\",\n  \"conversationId\": \"conv_789\",\n  \"idempotencyKey\": \"embed:file_abc:chunk_001\"\n}\n```\n\nAssume a queue message can be delivered more than once.\n\n---\n\n## R2 = artifact storage\n\nUse R2 for:\n\n- uploaded files\n- generated reports\n- raw transcripts\n- tool outputs\n- large intermediate results\n- images/audio/video\n- crawl results\n- code execution artifacts\n\nDo not put huge documents directly into Durable Object state or LangGraph state.\n\nBad:\n\n```json\n{\n  \"state\": {\n    \"entirePdfText\": \"500 pages...\"\n  }\n}\n```\n\nGood:\n\n```json\n{\n  \"state\": {\n    \"artifactRef\": \"r2://tenant-123/docs/file-456.txt\"\n  }\n}\n```\n\n---\n\n## D1 or Postgres = relational metadata\n\nUse D1 for smaller Cloudflare-native relational needs:\n\n- users\n- orgs\n- conversations index\n- run records\n- billing metadata\n- audit records\n- agent configs\n\nBut D1 is not always the right store for a large production agent platform.\n\nFor serious LangGraph deployments, Postgres is usually better because LangGraph/LangSmith already expect it for:\n\n- checkpoints\n- threads\n- runs\n- assistants\n- memory store\n- resumability\n\nSo a common split is:\n\n```text\nCloudflare D1:\n- edge-native app metadata\n- lightweight config\n\nPostgres:\n- LangGraph checkpoints\n- threads/runs\n- production agent state\n```\n\n---\n\n## KV = stale-tolerant config/cache\n\nUse KV for:\n\n- prompt templates\n- feature flags\n- model routing config\n- static public config\n- low-value cache\n\nDo **not** use KV for:\n\n- locks\n- counters\n- payment state\n- current conversation memory\n- read-after-write correctness\n- agent coordination\n\nKV is eventually consistent.\n\nThat means this is risky:\n\n```text\nWrite latest agent state to KV\nImmediately read latest agent state from another location\n```\n\nYou may get stale data.\n\n---\n\n## Vectorize = semantic memory/RAG\n\nUse Vectorize for:\n\n- document chunk search\n- long-term semantic memory\n- similar past conversations\n- user preference retrieval\n- knowledge base RAG\n\nPattern:\n\n```text\nR2:\n- source document text\n\nD1/Postgres:\n- chunk metadata\n- permissions\n- document ownership\n\nVectorize:\n- chunk embeddings\n- semantic search\n```\n\nDo not treat the vector DB as the source of truth.\n\n---\n\n## AI Gateway = model control plane\n\nPut model calls through AI Gateway.\n\nBenefits:\n\n- observability\n- model logs\n- retries\n- caching\n- rate limits\n- fallback\n- provider abstraction\n- cost visibility\n\nExample:\n\n```text\nLangChain.js / LangGraph\n   |\nAI Gateway\n   |\nOpenAI / Anthropic / Workers AI / other provider\n```\n\nProduction warning:\n\nBe careful with logging and caching. Prompts may include:\n\n- personal data\n- secrets\n- tool outputs\n- customer documents\n- internal business info\n\nSet redaction and caching rules intentionally.\n\n---\n\n# 4. How to make LangChain agents stateful\n\nBasic LangChain agents are not automatically production-stateful.\n\nA production agent needs state like:\n\n```ts\ntype AgentState = {\n  conversationId: string;\n  tenantId: string;\n  messages: Message[];\n  summary: string;\n  memories: MemoryRef[];\n  activeRunId?: string;\n  pendingApprovals: Approval[];\n  toolLedger: ToolCallRecord[];\n  budget: {\n    maxTokens: number;\n    maxToolCalls: number;\n    maxRuntimeMs: number;\n  };\n};\n```\n\nFor Cloudflare-native agents, store this in:\n\n```text\nDurable Object SQLite/state\n```\n\nFor LangGraph agents, store it in:\n\n```text\nLangGraph checkpointer, usually Postgres\n```\n\nFor large artifacts, store pointers:\n\n```ts\n{\n  type: \"artifact\",\n  uri: \"r2://tenant-123/runs/run-456/output.json\"\n}\n```\n\n---\n\n# 5. LangGraph/deepagents production pattern\n\nIf you are using LangGraph or LangChain deepagents, think in terms of **threads**, **runs**, and **checkpoints**.\n\n```text\nthread_id = durable conversation/workflow identity\nrun_id = one execution attempt\ncheckpoint = saved graph state after a node\n```\n\nA production flow:\n\n```text\n1. Client sends message.\n2. Worker authenticates.\n3. Worker resolves thread_id.\n4. Worker starts or resumes LangGraph run.\n5. LangGraph executes nodes.\n6. State checkpoints after each major node.\n7. Events stream back to client.\n8. If tool is risky, graph interrupts for approval.\n9. Human approves.\n10. Graph resumes from checkpoint.\n```\n\nThis is much more robust than:\n\n```text\nRun one giant agent loop in a single serverless request.\n```\n\n---\n\n# 6. Recommended architecture by maturity level\n\n## Level 1: Simple chat agent\n\nUse this if you are building a straightforward support/chatbot.\n\n```text\nWorker\n   |\nDurable Object per conversation\n   |\nLangChain.js agent\n   |\nAI Gateway\n   |\nVectorize/R2/D1\n```\n\nState:\n\n```text\nDurable Object:\n- recent messages\n- summary\n- active run\n\nVectorize:\n- semantic memory\n\nR2:\n- files and long transcripts\n\nD1:\n- metadata\n```\n\nGood for:\n\n- customer support bots\n- internal assistants\n- lightweight tool-using chat\n- low-latency edge apps\n\n---\n\n## Level 2: Stateful tool-using agent\n\nUse this when tools matter.\n\n```text\nWorker\n   |\nDurable Object\n   |\nAgent turn\n   |\nQueue for slow tools\n   |\nWorkflow for longer tasks\n   |\nAI Gateway\n```\n\nAdd:\n\n- idempotency ledger\n- tool timeouts\n- retry policy\n- per-run budgets\n- human approval for dangerous tools\n- structured logs\n\nGood for:\n\n- CRM assistant\n- calendar/email assistant\n- e-commerce assistant\n- ops assistant\n\n---\n\n## Level 3: Deep research/autonomous agent\n\nUse this for deepagents-style workloads.\n\n```text\nCloudflare Worker as gateway\n   |\nLangGraph/LangSmith Agent Server\n   |\nPostgres checkpointer\n   |\nR2 for artifacts\n   |\nVectorize or other vector DB\n   |\nQueues/workers for tools\n   |\nAI Gateway for LLM calls\n```\n\nGood for:\n\n- deep research\n- code agents\n- browser agents\n- long-running task agents\n- multi-agent/subagent systems\n- filesystem/workspace-heavy agents\n\nFor this level, I would usually **not** run the whole deepagent inside a Worker.\n\nUse Cloudflare at the edge, but run the heavy agent runtime in a container or managed LangGraph platform.\n\n---\n\n# 7. Long-running task pattern\n\nFor deep agents, do not do this:\n\n```text\nHTTP request starts\nAgent researches for 10 minutes\nHTTP request waits\nResponse eventually returns\n```\n\nDo this instead:\n\n```text\nPOST /runs\n   |\nCreate run_id\n   |\nStart Workflow or LangGraph run\n   |\nReturn run_id immediately\n```\n\nThen:\n\n```text\nGET /runs/:runId\n```\n\nor:\n\n```text\nGET /runs/:runId/events\n```\n\nor:\n\n```text\nWebSocket/SSE stream\n```\n\nState machine:\n\n```text\nqueued\nrunning\nwaiting_for_approval\ncompleted\nfailed\ncancelled\n```\n\nEach step should be resumable.\n\n---\n\n# 9. Human-in-the-loop pattern\n\nFor risky tools:\n\n- send email\n- make purchase\n- delete data\n- modify database\n- submit form\n- call external API with side effects\n\nUse an interrupt.\n\n```text\nAgent decides to send email\n   |\nStores proposed action\n   |\nMarks run as waiting_for_approval\n   |\nNotifies user\n   |\nUser approves/rejects\n   |\nAgent resumes\n```\n\nApproval record:\n\n```json\n{\n  \"approvalId\": \"appr_123\",\n  \"runId\": \"run_456\",\n  \"tool\": \"send_email\",\n  \"proposedInput\": {\n    \"to\": \"customer@example.com\",\n    \"subject\": \"Follow up\",\n    \"body\": \"...\"\n  },\n  \"status\": \"pending\",\n  \"createdAt\": \"2026-01-01T00:00:00Z\"\n}\n```\n\nDo not rely only on model text like:\n\n```text\n\"User approved this\"\n```\n\nStore real approval state.\n\n---\n\n# 10. Memory design\n\nA good agent memory stack has layers.\n\n## Short-term memory\n\nRecent conversation turns.\n\nStore in:\n\n- Durable Object\n- LangGraph checkpoint\n\nExample:\n\n```json\n{\n  \"recentMessages\": [\n    {\"role\": \"user\", \"content\": \"...\"},\n    {\"role\": \"assistant\", \"content\": \"...\"}\n  ]\n}\n```\n\n## Summary memory\n\nCondensed history.\n\nStore in:\n\n- Durable Object\n- D1/Postgres\n\nExample:\n\n```json\n{\n  \"summary\": \"The user is building a Cloudflare-hosted LangChain agent platform...\"\n}\n```\n\n## Semantic memory\n\nEmbeddings of important facts, documents, or past interactions.\n\nStore in:\n\n- Vectorize\n- external vector DB\n\nExample:\n\n```json\n{\n  \"text\": \"User prefers TypeScript examples.\",\n  \"embedding\": [0.123, -0.456],\n  \"metadata\": {\n    \"tenantId\": \"tenant_123\",\n    \"userId\": \"user_456\",\n    \"type\": \"preference\"\n  }\n}\n```\n\n## Artifact memory\n\nLarge files, reports, outputs, transcripts.\n\nStore in:\n\n- R2\n\nExample:\n\n```json\n{\n  \"artifactId\": \"artifact_123\",\n  \"uri\": \"r2://tenant-123/runs/run-456/report.md\"\n}\n```\n\n---\n\n# 11. Idempotency is non-negotiable\n\nProduction agents retry.\n\nQueues retry. Workflows retry. Users refresh. HTTP calls timeout. Models repeat tool calls.\n\nEvery side-effecting action needs an idempotency key.\n\nExample:\n\n```json\n{\n  \"tool\": \"send_email\",\n  \"idempotencyKey\": \"run_123:node_send_email:attempt_1\",\n  \"inputHash\": \"abc123\"\n}\n```\n\nBefore executing:\n\n```text\nCheck ledger:\n- Have we already executed this exact action?\n- If yes, return stored result.\n- If no, execute and record result.\n```\n\nTool ledger:\n\n```json\n{\n  \"idempotencyKey\": \"run_123:send_email:abc123\",\n  \"tool\": \"send_email\",\n  \"status\": \"completed\",\n  \"resultRef\": \"r2://tenant-123/tool-results/result-789.json\",\n  \"createdAt\": \"...\"\n}\n```\n\nThis prevents duplicate emails, duplicate purchases, duplicate database writes, etc.\n\n---\n\n# 12. Streaming design\n\nFor chat UX, stream tokens or events.\n\nFor deep agents, stream **events**, not just text.\n\nExample event stream:\n\n```json\n{\"type\": \"run_started\", \"runId\": \"run_123\"}\n{\"type\": \"model_started\", \"model\": \"claude-3-5-sonnet\"}\n{\"type\": \"tool_started\", \"tool\": \"web_search\"}\n{\"type\": \"tool_completed\", \"tool\": \"web_search\"}\n{\"type\": \"checkpoint_saved\"}\n{\"type\": \"message_delta\", \"text\": \"I found three relevant sources...\"}\n{\"type\": \"approval_required\", \"approvalId\": \"appr_456\"}\n```\n\nPersist events so the client can reconnect.\n\nEach event should have:\n\n```json\n{\n  \"runId\": \"run_123\",\n  \"threadId\": \"thread_456\",\n  \"eventIndex\": 42,\n  \"type\": \"tool_completed\",\n  \"timestamp\": \"...\"\n}\n```\n\n---\n\n# 13. Observability\n\nFor production, trace every run.\n\nAt minimum, propagate these IDs everywhere:\n\n```json\n{\n  \"requestId\": \"...\",\n  \"tenantId\": \"...\",\n  \"userId\": \"...\",\n  \"conversationId\": \"...\",\n  \"threadId\": \"...\",\n  \"runId\": \"...\",\n  \"workflowId\": \"...\",\n  \"model\": \"...\",\n  \"provider\": \"...\",\n  \"idempotencyKey\": \"...\"\n}\n```\n\nUse:\n\n- AI Gateway for model observability\n- LangSmith for LangChain/LangGraph traces\n- Cloudflare logs for Workers/Durable Objects\n- Queue dead-letter logs\n- Workflow status logs\n- custom audit tables\n- cost/token tracking\n\nTrack:\n\n- latency\n- token usage\n- model cost\n- tool failure rate\n- retries\n- approval waits\n- vector retrieval quality\n- hallucination reports\n- user feedback\n- run failure reasons\n\n---\n\n# 14. Security model\n\nProduction agents are dangerous because they can act.\n\nYou need:\n\n## Authentication\n\nValidate user/session at the Worker.\n\n## Authorization\n\nEvery tool must check tenant/user permissions.\n\nBad:\n\n```text\nModel says: \"Fetch document 123\"\nTool fetches document 123\n```\n\nGood:\n\n```text\nTool checks:\n- Does this user belong to this tenant?\n- Can this user access document 123?\n- Is this action allowed?\n```\n\n## Tool permissions\n\nGive each tool the minimum credentials needed.\n\n## Human approval\n\nRequire approval for irreversible actions.\n\n## Prompt injection defense\n\nNever let retrieved text override system/tool rules.\n\nTreat RAG/web content as untrusted data.\n\n## Audit logs\n\nRecord every side-effecting action.\n\n---\n\n# 15. Common mistakes\n\n## Mistake 1: Storing state in Worker globals\n\nBad for correctness.\n\nUse Durable Objects or external DB.\n\n---\n\n## Mistake 2: Using KV for agent memory\n\nKV is eventually consistent.\n\nUse Durable Objects, D1/Postgres, or LangGraph checkpoints.\n\n---\n\n## Mistake 3: Running a deepagent inside one HTTP request\n\nDeep agents are often long-running.\n\nUse Workflows, Queues, or LangGraph server.\n\n---\n\n## Mistake 4: Putting huge data in graph state\n\nStore references to R2/object storage instead.\n\n---\n\n## Mistake 5: No idempotency\n\nRetries will duplicate tool actions.\n\nUse a side-effect ledger.\n\n---\n\n## Mistake 6: No budgets\n\nAgents can loop forever.\n\nAdd limits:\n\n```json\n{\n  \"maxIterations\": 20,\n  \"maxToolCalls\": 30,\n  \"maxTokens\": 100000,\n  \"maxRuntimeMs\": 600000,\n  \"maxCostUsd\": 2.00\n}\n```\n\n---\n\n## Mistake 7: No model-call observability\n\nUse AI Gateway and/or LangSmith early.\n\nDo not wait until production incidents.\n\n---\n\n# 16.
